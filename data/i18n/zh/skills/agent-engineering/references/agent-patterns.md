# Agent 范式与 Harness

## Workflow 与 Agent

Workflow 适合步骤稳定、边界明确的任务。Agent 适合需要在运行时根据
观察结果选择工具、调整计划或恢复失败的任务。

判断标准：

- 步骤固定：用 workflow。
- 需要探索外部环境：用 Agent。
- 需要持续记忆或跨轮任务：用 Agent + memory。
- 需要多人协作角色：先确认职责是否可拆，否则不要上 multi-agent。

## 常见范式

### ReAct

结构是 Thought、Action、Observation 循环。适合工具调用和探索型任务。

关键点：

- 每次只选择一个动作。
- Observation 必须写回状态。
- 设置最大步数和失败退出条件。

### Plan-and-Solve

先规划，再按步骤执行。适合中等复杂度的开发、分析和迁移任务。

关键点：

- 计划要可检查、可拆分。
- 执行时允许更新计划，但必须说明原因。
- 每步结束要有可验证产物。

### Reflection

生成结果后进行自检、批评、修复。适合代码生成、文档、研究报告。

关键点：

- Reflection 不能只是复述，要对照验收标准。
- 失败案例要进入 memory 或测试集。
- 不要无限反思，设置迭代上限。

## Agent Harness 七层

1. 模型层：选择模型、温度、上下文窗口和成本策略。
2. 循环层：定义观察、思考、行动、停止条件。
3. 工具层：定义 schema、权限、超时、错误处理。
4. 记忆层：短期上下文、长期记忆、检索、任务笔记。
5. 人格层：角色、语气、领域约束、输出规范。
6. 通道层：用户、工具、日志、审查、后台任务的通信边界。
7. 自驱层：计划、恢复、任务队列、监控和升级策略。

最小实现不要一次做满七层。先实现模型、循环、工具和评估。
